Detailed Description

template<class F, class D>
class jpw_nld::fortlib::FitLM_Adapter< F, D >

Perform a nonlinear least-squares fit by passing a functor to the Levenberg-Marquardt algorithm.

Unlike its parent, FitLM, this class separates the model from the data being fit to it, using template parameters to specify both model and data in a typesafe fashion.

Template Parameters
D
F
D
A sequence data-container. (i.e. a 1-dimensional data structure).
It must contain the following members:
  • public INT_TYPE size() const
    Should return the size of the current dataset. Will be passed as the first arg to FitLM::operator()().
F
The fit-functor. Computes the function that you are fitting the data to.
Must have the following members:
  • public static const INT_TYPE N_PARAMETERS
    Will be static_cast to an unsigned integer type.

  • void operator()(int nData, const D& fitData,
    int nParams, fort_dvec_t curParams,
    fort_dvec_t deltas, fort_dmat_t fnJacob,
    int ld_fnJac, int actionCode)

    This is the fit-functor itself, which computes the model.
    The parameters are as as follows:
    • nData
      The number of data points to fit to. Will always be identical to fitData.size().
    • fitData
      This is the instance of D that was passed to FitLM_Adapter::operator()(). The contents will remain unaltered.
    • nParams
      The number of tunable parameters in the model.
    • curParams
      An nParams element long fort_dvec_t, containing the current set of tunable parameters that the Levenberg-Marquardt algorithm is testing.
      You do not need to perform any memory-management with this fort_dvec_t. That is all handled externally.
      DO NOT MODIFY its contents!
    • deltas
      A nData element long fort_dvec_t, pre-allocated externally for you. (Therefore, you do not need to perform any memory-management operations on it.)
      Must be filled with difference between the fitData and the model. See below for details of how you should compute it.
    • fnJacob
      An nParams by ldfjac matrix, flattened into a fort_dmat_t. You do not need to perform any memory-management on it, as it has been pre-allocated externally for
      Must be filled with the Jacobian matrix of the model being fit to, computed using curParams. See below for details of how to do this.
    • ld_fnJac
      The "long-dimension" of fnJacob. Should always be identical to nData.
    • actionCode
      An integer code for what the functor should compute.
      • When equal to 1, deltas must be computed and filled in, while fnJacob will remain unaltered.
      • When equal to 2, deltas must be left alone and fnJacob must be calculated and filled in.

The typedefs fort_dvec_t and fort_dmat_t are defined in "FORTTypes.h".


Writing a Fit-Functor

Implementing the fit-functor requires some knowledge of how the Levenberg-Marquardt fits the data to the model. The documentation of FitLM goes into the full details.

Although they're passed to your fit-functor, 3 of the arguments are redundant:

  • nParams
    Since the model is likely a mathematical function that you wrote on paper, you already know the number of parameters.
    Furthermore, this argument should always have the same value as the inherited member FitLM::m__nparams.
  • nData
    As noted above, this argument will always be identical to fitData.size(). You could use it instead of calling fitData.size(), thereby optimizing away the function call. Or, if you prefer, you could safely ignore it and hope that the compiler optimizes your fitData.size() calls.
  • ld_fnJac
    As stated earlier, the value of this argument will be identical to nData. It will also be identical to the inherited member FitLM::m__ndata.
    So, if you wish, you could ignore both this and nData and just use FitLM::m__ndata instead.

Computing “deltas”

Your fit-functor, therefore, should perform the following set of steps for each element of fitData:

  • Find the independent variable(s), 'x[j]', corresponding to 'fitData[j]';
  • Using the parameter values from curParams, compute 'f_j', the value of the fit-function evaluated at 'x_j';
  • Set:
    deltas[j] = fitData[j] - f_j
  • Repeat for all 'j'.

Note that, if your model is a multivariate function, you'll need to use a list of pairs or tuples as the 'x[j]' instead of using a list of scalars.
Remember: You only compute deltas when the argument actionCode==1. Otherwise, do not touch deltas at all.

Computing “fnJacob”

The algorithm also requires a matrix of the model's partial-derivatives. In the context of minimization algorithms, this matrix is known as a "Jacobian matrix." (However, for fitting data to a function, it's not really a matrix, since it's 1-dimensional.) To find the Jacobian for your model, you will need to do the following partial-derivatives on paper:

Notice that you are taking the partial-derviatives with respect to the parameters, because in the context of fitting to a model, it's the parameters that change (not the independent variable).

You now have nParams equations. You will need to compute them at the same that you used to compute the model itself:

This gives you a nParams by nData matrix.

The fit-functor will also be computing the Jacobian. Here are the steps that it should perform:

  • Use the same independent variable(s), 'x[j]', corresponding to 'fitData[j]', that you used to compute the 'deltas[j]'.
    • If your 'x[j]' is some fixed, pre-build array, you're done.
    • If you have to construct 'x[j]', construct it in the exact same order that you did when you computed 'deltas[j]'.
    • And, just in case it's not clear,
      The order of the 'x[j]' is important!
  • You will have nParams derivatives of your model.
    • Let's label the ith derivative:
      df_dpi(double y, double* paramVec)
      Remember: This is the partial-derivative of your model with respect to the ith parameter.
    • You don't need to write a separate function for each of the derivatives. In fact, you probably shouldn't for performance issues. We're only writing it as a function to make this explanation clearer.
  • Repeat the following for each of your tunable parameters:
    • Set the integer variable, 'i_param', to the index of the parameter whose derivative you're about to evaluate.
    • Compute the value 'df_dpi_x_j = df_dpi(x[j], curParams)'
      • Remember: you need to pick the correct 'df_dpi(...)', the one that is the partial-derivative with respect to the i_paramth parameter.
      • You will compute the i_paramth partial-derivative using 'x[j]' and the entire array, 'curParams'.
        • Again: you're not using only 'curParams[i_param]', you'll likely need all of the elements of 'curParams'.
        • In fact, you may have differentiated away the i_paramth parameter.
    • Set:
      fnJacob[j + ld_fnJac*i_param] = df_dpi_x_j
      Note the indexing used. Remember that fnJacob has been flattened from a nParams by ld_fnJac 2-D matrix into a 1-D array. (While ld_fnJac might equal nData, you shouldn't count on that.)
  • Repeat for all 'j'.

The same caveat for multivariate functions applies here: you'll need to use a list of pairs or tuples as the 'x[j]' instead of a scalar array.
Remember: You only compute fnJacob when the argument actionCode==2. Otherwise, do not touch fnJacob at all.

Wrapper Function, passed to FitLM::operator()().

You will never call this function directly.

This static member function calls the FitFunctor_t and Data_t objects that were passed to the last FitLM_Adapter::operator()() call. Unfortunately, it works by using a static member field, making FitLM_Adapter and all of its subclasses inherently non-re-entrant.

The details are a bit more complex, due to the template parameters. Every instantiated template class has its own copies of the template's static member fields. They also have their own, separate implementations of each static member function. So, each time you template-instantiate FitLM_Adapter with a different functor type (or data container type), you've created a completely separate version of the fit_function_adapter. That means you can use object-instances of these two different template-instantiations simultaneously. For example:

class TrigModel;
class LogarithmicModel;
class FitLM_Adapter<TrigModel, std::vector> SpecialTrigFit
{
// Declaration/Definition Omitted for Clarity
// :
// :
};
FitLM_Adapter<LogarithmicModel, std::vector> logfit;
FitLM_Adapter<TrigModel, std::vector> trigfit;
SpecialTrigFit child_trigfit;

You could use logfit and trigfit at the same time (well, if the FORTRAN function, lmder_, is re-entrant). In contrast, you cannot use trigfit and child_trigfit simultaneously.

These are the issues that make this class potentially non-re-entrant.

Perform a nonlinear least-squares fit.

Perform a nonlinear least-squares fit using the Levenberg-Marquardt algorithm.

Most of the arguments are identical to those in FitLM::operator()() as is the return value.

Parameters
theModelFunctor object of type F. A pointer to it is held for the duration of this function call.
theDataData container object of type D. A pointer to it is held for the duration of this function call. Will be passed to theModel by the LM algorithm.
params0The vector of initial values of the tunable parameters. Element order is never changed. (I.e. - theModel will receive it in the same order that you pass it here.)
errtolThe tolerance for , the sum of the least-squares. If the change in from one iteration to the next is less than errtol, the algorithm stops.
ptolThe tolerance for the set of parameters. Once the (magnitude of the) parameter vector changes by less than this quantity, the algorithm stops.
factorThe initial maximum step size. A typical value is in the range [0.1, 100.0]. If factor==0, a value of factor=100.0 is used instead.
maxiterThe maximum number of times that the algorithm calls theModel to recompute the error-vector
