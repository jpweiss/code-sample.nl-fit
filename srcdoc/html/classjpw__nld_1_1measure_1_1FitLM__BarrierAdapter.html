Detailed Description

template<class F_POL_T = policy::Full>
class jpw_nld::measure::FitLM_BarrierAdapter< F_POL_T >

Child class of the FitLM_Adapter, using the BarrierModel and PersistenceMap classes.

The F_POL_T class is the same policy type taken by the BarrierModel class.

Note: Because this header uses "BarrierModels.h", translation units #including this header should also #include "details/BarrierModels.tcc".

typedef void(* fit_function_ptr_t)(fort_ivar_t, fort_ivar_t, fort_dvec_t, fort_dvec_t, fort_dmat_t, fort_ivar_t, fort_ivar_t) [inherited]

Function pointer type for the fit function.

This is the function to minimize. It should produce a vector of mm floating-point values.

Since the Levenberg-Marquardt algorithm minimizes this function, if you wish to instead fit a model to data, you must somehow build the data into the fit function. Unfortunate, but we're stuck with the design of the underlying FORTRAN code.

The parameters are, in order:

Parameters:
mminteger
The number of equations (or the number of data points to fit to). Will be the same as the value passed to the first argument of operator()(int, dvector_t&, double, double, int, double).
nninteger
The number of variables in the equations (or parameters in the model being fit).
xdouble
An nn element vector of the variables (or for fitting to a model, the values of the parameters), to use to compute deltas or fnJacob.
deltasdouble
An mm element vector. Should be filled with the value of the mm equations (or the model minus the data), computed using x.
(Each deltas[j] must obviously be computed with x[j].)
fnJacobdouble
An nn by ld_fnJac matrix, flattened into an array. Should be filled with the Jacobian of the mm equations, computed using x.
The matrix has been flattened into fnJacob[j + mm*i], where i is the index representing a specific equation and j represents the index used with the array, x. Be sure to compute each of the mm elements of fnJacob with the correct corresponding element of x !
ld_fnJacinteger
The long-dimension of fnJacob. Will be identical to the value passed to the constructor argument, ndata_max. Should therefore be identical to mm (unless you're doing something weird).
actionCodeinteger
Control flag. When set to 1, the function should compute deltas and leave fnJacob unaltered. When set to 2, the function should leave deltas alone and compute fnJacob. The function can set this to a negative number to terminate minimization, but shouldn't modify it otherwise.
Consider using the FitFnAction_t enum to check the value of this argument, rather than the nondescript integer values.

enum FitFnAction_t [inherited]

Action-codes passed by the Levenberg-Marquardt code to the fit-function.

Enumerator:
ComputeFunction 

Tells the fit-function to compute the model being fit (or function being solved).

The deltas argument to the fit-function should be filled in, while the fnJacob argument should be left unaltered.

ComputeJacobian 

Tells the fit-function to compute the Jacobian matrix.

In this case, the deltas argument to the fit-function must be left unaltered, while the fnJacob argument will be filled in.

) [inline, static, protected, inherited]

Wrapper Function, passed to FitLM::operator()().

You will never call this function directly.

This static member function calls the FitFunctor_t and Data_t objects that were passed to the last FitLM_Adapter::operator()() call. Unfortunately, it works by using a static member field, making FitLM_Adapter and all of its subclasses inherently non-re-entrant.

The details are a bit more complex, due to the template parameters. Every instantiated template class has its own copies of the template's static member fields. They also have their own, separate implementations of each static member function. So, each time you template-instantiate FitLM_Adapter with a different functor type (or data container type), you've created a completely separate version of the fit_function_adapter. That means you can use object-instances of these two different template-instantiations simultaneously. For example: 

 class TrigModel;
 class LogarithmicModel;

 class FitLM_Adapter<TrigModel, std::vector> SpecialTrigFit
 {
     // Declaration/Definition Omitted for Clarity
     // :
     // :
 };

 FitLM_Adapter<LogarithmicModel, std::vector> logfit;
 FitLM_Adapter<TrigModel, std::vector> trigfit;
 SpecialTrigFit child_trigfit;

You could use logfit and trigfit at the same time (well, if the FORTRAN function, lmder_, is re-entrant). In contrast, you cannot use trigfit and child_trigfit simultaneously.

These are the issues that make this class potentially non-re-entrant.

Perform a nonlinear least-squares fit.

Perform a nonlinear least-squares fit using the Levenberg-Marquardt algorithm. See the c'tor for some related info.

Parameters:
mmThe number of data points used in the fit. It must be in the range [nparm, ndata_max], inclusive, where ndata_max and nparm are two of the parameters you passed to the c'tor.
You will almost always use the same value here as you did for ndata_max in the c'tor.
xvA vector of initial parameter values. On return, it will contain the best-fit values of the parameters.
errtolThe tolerance for . If the change in from one iteration to the next is less than this value, the algorithm stops.
ptolThe tolerance for the set of parameters. If the magnitude of the parameter vector changes by less than this value, the algorithm stops.
factorThe initial maximum step size. A typical value is in the range [0.1, 100.0]. If factor==0, a value of factor=100.0 is used instead.
maxiterThe maximum number of times that the algorithm recalculates deltas [see fit_function_ptr_t] before stopping. Not really the number of iteration, but close enough. If the initial value of maxiter==0, a value of maxiter=100*(ndata_max+1) is used.
Returns:
The error codes, which are the same that the FORTRAN routine, lmder_(), returns in its info parameter.

Most of the arguments are identical to those in FitLM::operator()() as is the return value.

Parameters:
